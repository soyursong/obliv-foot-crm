## 2. 3-key 축별 공식 source-of-truth 매핑

| canon 3-key 축 | 1차 검증 소스 | 비고 |
|---------------|--------------|------|
| **코드** | **A(표준코드/품목기준코드)** + **B(급여 EDI코드)** | 우리 `claim_code` 체계와 직접 정합. DA §14 namespace 그대로 |
| **상품명** | **A(한글상품명)** | 표준코드 master가 정규 상품명 보유 |
| **성분명** | ⚠ **A에 성분명 컬럼 부재** → **E(식약처 허가정보 주성분)** 또는 **F(ATC)** 보조 enrich 필요 | AC-2 매칭설계에서 보조축으로 결정 |
| (급여/비급여) | **B** | 검증 자체엔 불필요하나 배지 부가정보로 활용 가능 |

> **핵심 한계 1건**: HIRA 약가마스터(A)는 상품명·코드·규격은 강하나 **성분명 컬럼이 명시적이지 않다**(namematch v3 확인: 컬럼=상품명/업체명/규격/품목기준코드/대표코드/표준코드). 따라서 3-key 중 **성분명 축의 공식 검증은 식약처 허가정보(E) 보조가 필요**. 단 1차 검증(상품명+코드)은 A 단독으로 성립하며, 성분명은 2차 보강. → AC-2에서 "코드 정확매칭 1차 / 성분 보조 enrich" 2단으로 설계.

## 3. 권고 (1안)

### ✅ 권고 소스 = **A(약가마스터 의약품표준코드, 15067462 공개파일) + B(약제급여목록표, 기존 `hira_insurance_sync.mjs`)** 조합. 성분명 축은 **E(식약처 허가정보)를 2차 보조**.

**근거**
1. **코드체계 정합**: canon의 '코드'(=우리 `claim_code`)가 EDI/표준코드(HIRA) 체계다. 식약처(D·E)는 허가번호·품목코드라 우리 claim_code와 **불일치** → 코드 대조의 1급 source는 반드시 HIRA.
2. **중복 적재 금지 준수**: HIRA-MAP(T-20260617)이 이미 A·B를 채택·검증했고 DA가 claim_code namespace를 확정(§14). 본 티켓이 같은 A·B를 쓰면 **단일 소스·코드체계 공유**가 자동 성립. 다른 소스 선택 시 이중 적재·코드 충돌.
3. **외부의존 리스크 최소**: A·B는 **무키 공개파일 다운로드** → API 키 관리·rate limit·비용·외부 API 장애(AC-5 fallback 부담) 0. 식약처 OpenAPI(D·E)는 키·쿼터·외부 호출 의존 → 운영 리스크↑.
4. **canon 정합**: A·B 모두 코드 정확매칭 기반 → drug_identity_rule(auto-merge 금지·코드 우선)과 정합. 퍼지매칭 불필요.

**보조(성분명 축, 선택)**: E(식약처 허가정보)로 주성분을 enrich. 단 외부 API 키가 필요하므로, **성분명 검증을 1차 차단요소로 삼지 않고**(AC-6 비차단) "상품명+코드 일치 시 verified, 성분은 부가표기" 수준으로 시작 권고. 성분 공식대조를 강화하려면 E 도입을 별도 판단(키 발급=형 확인 게이트).

## 4. 다음 단계 (AC-2~AC-6, 본 FOLLOWUP 이후)

- **직렬화 대기**: AC-3(검증상태 캐시 스키마)은 prescription_codes 클러스터(RXCODES-WRITE-RLS in_progress, AUTO5 3-key 재검증 FIX-REQUEST 진행) **정착·종료 후** 착수. 스키마 필요 시 ADDITIVE만 + data-architect CONSULT 선행 → supervisor DDL-diff.
- **AC-2 매칭설계**: 코드(표준코드/EDI) 정확매칭 1차, 코드 부재 시 (상품명) 보조, **퍼지/용량표기 자동연결 금지**(모호=미확인=사람확인). 성분명은 E 보조 enrich(2차).
- **AC-5 fallback**: A·B는 사전 적재(파일)라 런타임 외부 호출 없음 → 외부 장애 fallback 부담 자체가 작음. 만약 성분 enrich로 E API 도입 시 그 호출에 graceful degrade(unverified) 적용.
- **결정 필요(planner/형)**: 성분명 공식 검증을 위해 **식약처 OpenAPI(E) 키 발급**을 진행할지. 미진행 시 1차 검증=상품명+코드(HIRA)만, 성분=내부값 표기.
