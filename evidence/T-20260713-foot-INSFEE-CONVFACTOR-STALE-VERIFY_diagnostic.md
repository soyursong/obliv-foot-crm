**현재값·갱신시점:** 코드·시드 기준 **89.4 / year 2024**. `89.4→93.6/94.1/95.6` 로 갱신한 마이그레이션은 레포 전체에 **0건** (grep 확인). **심평원 자동연동/cron 없음** — 2026-05-04 DEFAULT 설정 이후 **한 번도 갱신 안 됨**. (prod `clinics` 행 실제값은 별도 SELECT 확인 권고 — 단, fallback이 89.4라 컬럼 NULL이어도 89.4로 귀결.)

**★ 산정경로:** foot은 급여를 **자체 산정**. `base(수가) = ROUND(hira_score[상대가치점수] × hira_unit_value[환산지수])` — `copayCalc.ts:120`, RPC `...:189`. 심평원 EDI/수가표를 그대로 받는 방식 **아님**. ediExport.ts는 이 산출 결과(service_charges)를 청구로 내보냄. ∴ **환산지수 오류 = 수가 오류를 그대로 유발**.

**단일 소스 확인:** 산출 로직은 서버 RPC `calc_copayment`(DA SSOT, TK-ACC-2) + 클라 미러 `copayCalc.ts`(calcCopaymentLocal, useInsurance.ts·Chart2InsuranceCalcPanel·InsuranceCopaymentPanel 소비) 2벌뿐. 둘 다 동일한 89.4 fallback 공유 → 별도 divergent 로직은 없음(단 아래 §related 주의).

## AC2 — 환산지수 정확성 (고시 원문 대조)

의원급 점수당 단가(환산지수), 보건복지부·심평원 고시 기준:

| 연도 | 실제(고시) | 코드값 | 판정 |
|------|-----------|--------|------|
| 2024 | **93.6원** | 89.4 (2024로 라벨) | ❌ 라벨·값 모두 불일치 |
| 2025 | **94.1원** | (없음, 89.4 유지) | 제보 주장 94.1 = **맞음** |
| 2026 | **95.6원** | (없음, 89.4 유지) | 제보 주장 95.6 = **맞음** |

- 제보 주장값(2025 94.1 / 2026 95.6) **모두 정확** (medipana 329301, paxmedicalnews 840, 보건복지부 보도자료).
- 2026 인상률 평균 1.93%, 의원 94.1→95.6.

**결론: 89.4는 stale 확정.** 2026 기준 최소 **6.94% 낮음** ((95.6−89.4)/89.4).

## AC4 — 65세 노인정액제

**현행 코드 (RPC `20260504000000_...:197` + 클라 `copayCalc.ts:126`):**
```
elderly_flat AND base ≤ 15,000  → copay = MIN(1,500, base)   # 정액
그 외(base > 15,000)            → ELSE 분기 → rate 0.30 (30%)  # getBaseCopayRate('elderly_flat')=0.30
```

**실제 제도 (심평원 노인외래정액제, 2018~ 구간별 정률):**
| 총진료비(수가) | 실제 본인부담 | 코드 | 판정 |
|---------------|--------------|------|------|
| ≤ 15,000 | 1,500 정액 | 1,500 정액 | ✅ 일치 |
| 15,000~20,000 | **10%** | 30% | ❌ 과다 |
| 20,000~25,000 | **20%** | 30% | ❌ 과다 |
| > 25,000 | 30% | 30% | ✅ 일치 |

**판정: 구간제 미반영.** ≤15,000 정액 tier와 >25,000 30% tier만 우연히 맞고, **중간 10%/20% 두 구간 누락** → 15,000~25,000원 구간 65세 환자에게 30% 과다 청구.
- 예: 수가 18,000 → 실제 1,800(10%) / 코드 5,400(30%) → **+3,600 과다 수납**.
- 예: 수가 22,000 → 실제 4,400(20%) / 코드 6,600(30%) → **+2,200 과다 수납**.
- ⚠ 상호작용: 환산지수 stale(base 6.9%↓)이 경계값(15,000/20,000/25,000) 인접 건을 **하위 구간으로 오분류**시킬 수 있음 → 두 결함 복합 영향.
- 관계: 단일 분기(RPC line 197, copayCalc.ts line 126). 수정 시 두 파일 동시.

## AC5 — 권위경계 & 다음 액션

**소유권:**
- **환산지수 값(숫자)** = 심평원/보건복지부 고시 = **external SSOT**. foot은 `clinics.hira_unit_value`에 저장·적용만.
- **산식**(base=score×unit, 정액/정률 분기) = **DA SSOT** (`calc_copayment`, TK-ACC-2).
- ∴ 환산지수 값 갱신·65세 구간제 반영 **모두 급여 산식 변경** → 신규 컬럼/enum이 아니어도 **DA CONSULT 선행 필수** (§S2.4 데이터 정책 자문 게이트).

**소급 범위:** `service_charges`가 charge 시점 `hira_unit_value`·`customer_grade`·`copayment_rate` **스냅샷 보존** → 감사·재계산 추적 가능. 단 소급 재계산은 **결제완료·EDI전송 건**을 건드림 → 공단 재청구 + 환자 정산(추가징수/환급 양방향) → **대표·현장·회계 confirm 게이트 + 법무 검토** 필요. 권고: **forward-fix 우선, 소급은 별도 판단**.

**후속 FIX 스코프 제안 (실행 아님 — planner→DA CONSULT 후 티켓화):**
1. [DA-owned] 환산지수 소스 정비: 하드코딩 89.4 fallback(6개소) → 연도별 마스터/현행값. 값=정책, 매년 갱신 프로세스 필요.
2. [DA-owned] `calc_copayment` 65세 노인정액 **구간 정률(1,500/10%/20%/30%)** 반영.
3. `copayCalc.ts` ↔ RPC 동기화 — **COPAYCALC-CLIENT-ALIGN(blocked) 해소와 묶기** (client/server divergence 위험).
4. `clinics.hira_unit_value` prod 현재값 SELECT 확인 + 매년 갱신 거버넌스 수립.
5. 소급 영향 정량(service_charges 재계산 시뮬) = 별도 티켓 + 대표 게이트.

**관련 축 (divergent 수가 로직 점검):**
- SALESDOCTOR-INS-SPLIT / RECEIPT-ITEMIZED-INSURANCE-SPLIT / BILLDOC-INSSPLIT — 모두 `calc_copayment` 산출(base/copay/covered split) **단일 소스 소비** → divergent 로직 아님(다행). 단 6.9% 과소가 **전표·영수증·의사매출집계에 그대로 전파**됨.
- COPAYCALC-SERVER-NULLFIX(20260629190000) — 89.4 fallback 그대로 유지(null 처리만 수정, 값 미변경).
- COPAYCALC-CLIENT-ALIGN(blocked) — client `copayCalc.ts`가 server와 별도 파일 → **미해소 시 divergence 잔존 위험**. FIX 시 함께 처리 필요.
