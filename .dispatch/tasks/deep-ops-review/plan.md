# 현장 오퍼레이션 심층 반복 테스트

실제 클리닉 운영을 상세히 시뮬레이션. 각 역할이 하루에 50~100회 반복하는 작업을 기준으로 불편점 발굴.

- [x] TM팀 반복 테스트 (5회): 5건 연속 예약 성공. 수정/취소 버튼 없음, 유입경로 필드 없음, 일마감 빈 화면
- [x] 코디 오전 시나리오: 대시보드 칸반 확인, 상태 이동 UX 불명확, 이동이력 자동로깅 양호
- [x] 코디 피크타임 시나리오: 빈 시술실 15개 공간낭비, 마취경과 알림 부재, 우선순위 가이드 부재
- [x] 상담실장 반복 테스트 (5회): 시술 추가 다이얼로그 확인 (7개 시술), 검색/필터 없음, 과거 이력 미표시
- [x] 시술사 관점: 시술실 필터 없음 (15개 전체 표시), 시술완료 버튼 미확인
- [x] 데스크 결제 반복 (5회): 결제/수납 UI 완전 부재, 일마감 미구현
- [x] 고객 반복 (3회): 모바일 UI 양호, 개인정보 동의 모달 UX 이슈, English 지원 확인
- [x] 태블릿(1024x768) 전체 플로우: 시술 12번 이후 잘림, 완료 컬럼 스크롤, 일마감 빈 화면
- [x] 이슈 종합 + 액션아이템 작성 → output.md (6 Critical, 10 Medium, 7 Low/Good)

> **동시성 테스트 결과 (별도 실행):**
> - CRITICAL: 두 고객 동시 체크인 시 대기번호 #007 충돌 (동일 번호 부여) — next_queue_number RPC가 lock 없이 동시 호출됨
> - HIGH: 고객 체크인 후 코디 대시보드에 5초 후에도 Realtime 미반영
> - "명명" 이중접미사 버그 여전 (Lovable이 WaitingScreen 최신 코드 미반영)

테스트 시 반드시:
1. 매 액션마다 스크린샷
2. 스크린샷을 Read tool로 직접 확인하여 시각적 이슈 분석
3. 클릭 수, 시간, 불편함을 구체적으로 기록
4. "이게 하루 100번이면?" 관점으로 평가

**IMPORTANT: 동시 다중 브라우저 테스트**
- browser.newContext()로 여러 브라우저 컨텍스트를 동시에 열어서 테스트
- 예: 고객이 체크인하는 동시에 코디가 대시보드 보는 상황
- 예: TM이 예약 등록하는 동시에 코디가 같은 고객을 수동 등록하는 충돌
- 예: 두 명의 코디가 동시에 같은 환자를 다른 상담실로 드래그하는 상황
- 예: 고객이 대기화면 보는 동안 코디가 상태를 변경 → Realtime 반영 확인
- 반드시 Promise.all로 동시 실행하여 레이스 컨디션/동시성 이슈 발견
